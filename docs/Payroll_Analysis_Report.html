1. Executive overview

This folder contains the complete source and runtime data for a late-1990s desktop Payroll & Statutory Compliance application called FAS Payroll. It was built in Sybase PowerBuilder 5.0 on top of the PFC (PowerBuilder Foundation Classes) framework, and the data lives in a Watcom SQL / Sybase SQL Anywhere database file (FASMANGO.DB — about 1.38 GB once unzipped). The schema owner is min.

Functionally the application implements the full payroll cycle for an Indian organization: employee & department masters, payroll-period setup, attendance capture, advance & recovery processing, payroll calculation (including 9 user-defined salary components), full statutory deductions (EPF / FPF / EDLI / ESI / PPF / LWF / Profession-Tax / Income-Tax-TDS), pay-slip and salary-register printing, and a complete set of statutory return forms. Each financial transaction also posts a voucher into a sister Financial-Accounting (FA) module via a stored procedure f_post_fa.

How the file format was decoded: Every .PBL begins with the literal header HDR*PowerBuilder followed by the byte sequence 05 00 (PowerBuilder version 5.0) plus a label such as “FAS : Payroll System ( Window - 1 )”. Inside, each object's source is stored as plain ASCII text padded with NUL bytes and broken into 512-byte blocks separated by DAT* markers; converting NULs to newlines and stripping the markers yields readable PowerScript that can be parsed with regex. The FASMANGO.DB file contains the literal copyright string WATCOM International Corp., Copyright (c) 19... at offset ~250, confirming it as a Watcom SQL / SQL Anywhere physical database.

Within each row, the column suffix conventions are:

  • _ix — an index/surrogate primary or foreign key (always decimal(0), allocated by uf_nextval())
  • _cd — a short alphanumeric code visible to the user (e.g. emp_cd)
  • _title — the descriptive label for a code/index
  • _dt — date
  • _amt — monetary amount (always decimal(2))
  • _ba — "base amount" used as the basis for a percentage calculation
  • _rate — percentage or per-day rate
  • _no — a registration / cheque / document number string
  • _flg — a Y/N flag (char(3))
  • empe_* vs empr_* — employee's vs employer's portion of a statutory contribution
  • min_uid, min_mid, min_stat — record-level audit / status (see Section 15)

Project prefix FAS PY means “FAS Payroll”; FAS is the product/vendor brand. The database file name FASMANGO suggests the deployment was made for a customer or site called “Mango”.

Standard window layout (MDI sheet pattern). Every entry window contains four standard controls, already wired up by the w_master_entry_sheet_* ancestor:
  1. dw_select — a small DataWindow at the top for the user to enter search criteria (company, period, employee…)

7. Database schema (full)

Recovered by parsing the column=( ... dbname="table.col" type=... ) blocks inside every DataWindow source. The schema owner / SQL Anywhere user is min; every fully-qualified reference in the source code uses the form min.<table>. There are 12 application tables plus a public view vti_py_outst.

Type notes:
• decimal(0) columns ending in _ix are surrogate keys (long integers, zero-decimal storage)
• decimal(2) is used for all monetary amounts (rupees, two-decimal precision)
• char(N) is fixed-length CHAR (SQL Anywhere stores trimmed but the API pads on read)
• date is a normal SQL date (no time component)
• The Watcom SQL engine treats trailing whitespace as significant in equality unless "-b" is set

10. Salary calculation pipeline

The entry-point function is wf_calc_pay( arg_dw, arg_row ) in FASPYW1.PBL. It is the single orchestrator called once per employee row when the user clicks Calculate on the “Pay-Slip Generation” window. The body of wf_calc_pay verbatim reads:

// Main calculation sequence
wf_init( arg_dw, arg_row )                     // all init vars IX & FKs

// calculation for each user-defined component
wf_calc_comp_1( arg_dw, arg_row )
wf_calc_comp_2( arg_dw, arg_row )
wf_calc_comp_3( arg_dw, arg_row )
wf_calc_comp_4( arg_dw, arg_row )
wf_calc_comp_5( arg_dw, arg_row )
wf_calc_comp_6( arg_dw, arg_row )
wf_calc_comp_7( arg_dw, arg_row )
wf_calc_comp_8( arg_dw, arg_row )
wf_calc_comp_9( arg_dw, arg_row )

wf_calc_othrs( arg_dw, arg_row )               // calculate Tiffin / "Others" allowance
wf_calc_lwf( arg_dw, arg_row )                 // Calculate L.W.F.
wf_calc_emp_tot_earn( arg_dw, arg_row )        // Total earning
wf_calc_ptax( arg_dw, arg_row )                // Profession Tax
wf_calc_itax( arg_dw, arg_row )                // Income Tax (TDS)
wf_calc_pf_fpf( arg_dw, arg_row )              // EPF + Family Pension Fund (merged because of shared ceiling)
//wf_calc_ppf( arg_dw, arg_row )               // PPF (currently disabled in source)
wf_calc_esi( arg_dw, arg_row )                 // ESI
wf_calc_emp_tot_ded( arg_dw, arg_row )         // total deductions
wf_calc_emp_tot_pay( arg_dw, arg_row )         // total pay (= earnings - deductions)
wf_calc_oth_adj( arg_dw, arg_row )             // other adjustments
//wf_calc_adv_ded( arg_dw, arg_row )           // advance deductions (commented out â€” moved
                                               // into adv-deduction screen)
wf_calc_emp_net_pay( arg_dw, arg_row )         // net pay = total pay + oth_adj

return ( 1 )

The order matters. Earnings (comp_1..9 and "others") must be summed into emp_tot_earn before P.Tax / I-Tax / EPF / ESI run, because all of those use the total earnings as part of their base calculation. The deductions then accumulate into emp_tot_ded; emp_tot_pay = emp_tot_earn - emp_tot_ded; emp_net_pay = emp_tot_pay + oth_adj.

Why wf_calc_pf_fpf is one function: in the EPF Act, 12% of the basic+DA up to a ceiling goes into the EPF account, but a slice of the employer's 12% is diverted into the Family Pension Fund (FPF, now EPS). The ceiling and the EPF/FPF split are inter-dependent, so your father merged them into one function rather than two separate ones. The source comment confirms: "calcutlation of PF & FPF merged because of interrelation of ceiling limit".

12. Financial-Accounting (FA) integration

This payroll module does not stop at calculating salary — it also generates a journal voucher in the sister Financial-Accounting module for every advance, payment, receipt, advance-adjustment and salary run. The integration code lives in the window-level functions wf_post_fa, wf_kha_post_fa, wf_drv_post_fa, wf_del_fa and the helper wf_fa_detl_post.

12.1 Posting flow

  1. Allocate a fresh voucher index: ld_fa_ix = uf_nextval() (only on INSERT; UPDATE keeps the existing one).
  2. Look up the standard posting accounts from ds_df_fa_post_k (a non-visual DataStore loaded at startup) using identifiers like 'PB_IX', 'ADVANCE', 'PAYMENT' — this is how the configuration knows which account head to debit/credit for each kind of voucher.
  3. Populate the structure s_fa (header) and an array of s_fa_detl (details). The header carries: fa_ix, co_ix, site_ix, pb_ix, ref_no, fa_date, narration.
  4. For each detail line set: fa_ix, fa_detl_ix, ac_ix, drcr (D or C), amt (negative if Credit, positive if Debit), print_summary, cq_no, slno, narration.
  5. Call f_post_fa( task, fa_ix, lstr_fa, lstr_fa_detl[] ) — a global function (lives in the FA module) that actually does the INSERT/UPDATE on tr_fa and tr_fa_detl.

13.3 The big employee/balance lookup query

The longest single SELECT recovered from FASPYD2.PBL drives the “Leave Information” entry window. It picks the single most recent df_py_leave row that is still effective on the period’s end-date for every employee — a clever use of correlated subqueries:

SELECT  emp_ix, emp_full_name,
        ( SELECT from_dt
            FROM   min.df_py_leave p
            WHERE  p.emp_ix = df_py_emp.emp_ix
              AND  from_dt = ( SELECT max(from_dt)
                                 FROM  min.df_py_leave s
                                 WHERE s.emp_ix = p.emp_ix
                                   AND from_dt <= ( SELECT prd_to_dt
                                                       FROM  min.df_py_prd
                                                       WHERE prd_ix = :arg_prd_ix ) )
        ) as effective_balance_date
FROM    min.df_py_emp
WHERE   codept_ix = :arg_codept_ix

This pattern (“the latest row whose from_dt is ≤ period end”) is repeated for every leave-balance column. In a modern rebuild this would become a single LATERAL JOIN or a window function (ROW_NUMBER() OVER (PARTITION BY emp_ix ORDER BY from_dt DESC)).

16. Modernization recommendations

If you are rebuilding this in a modern stack, here is a pragmatic checklist drawn from the gaps and patterns visible in the original code:

Schema

  • Replace every _ix decimal(0) surrogate key with BIGSERIAL (PostgreSQL) or IDENTITY (SQL Server) or auto_increment (MySQL). Drop uf_nextval().
  • Add proper FOREIGN KEY constraints (none are declared at the schema level today; FK is enforced only at the UI).
  • Replace char(N) with varchar(N) — the trailing-padding semantics of CHAR cause bugs.
  • Replace the _yn char(3) Y/N flags with boolean / BIT.
  • Add created_at TIMESTAMP, updated_at TIMESTAMP (currently missing).
  • Move the 9 user-defined components from per-department comp_1_title…comp_9_title into a child table component (codept_id, slot_no, title, default_rate) — that frees you from the 9-component limit.
  • Move the hard-coded P.Tax slabs into a tax_slab (state, scheme, from_amt, to_amt, tax_amt, effective_from, effective_to) table.
  • Move the hard-coded 2002-03-31 ESI cutoff into a configuration row, or just drop the historical branch.
  • Move the hard-coded "LWF in June + December" rule into lwf_schedule rows so different states (which all have different schedules) can be handled.

Calculation engine

  • Re-implement wf_calc_pay as a pure function: calculate_pay( employee, period, attendance, advances ) -> payslip. No DataWindow side-effects.
  • Make every statutory calculator (EPF, FPF, ESI, P.Tax, I.Tax, LWF, PPF) a separate strategy class. The choice of which strategies run is data-driven (one row in the configuration says "this department uses these statutes").
  • Persist the calculated payslip as JSON in addition to the structured columns — that future-proofs you against schema changes when statutes evolve.

Application architecture

  • Replace the PowerBuilder MDI client with a web app (React / Vue / Angular front + REST or GraphQL API).
  • The w_master_entry_sheet_04 "select-and-edit" pattern maps almost 1:1 to a generic CRUD page with a search panel + data table + edit drawer/modal. Build that page once as a reusable component.
  • Replace the setmicrohelp footer message with toast notifications; keep the message wording (it’s already user-tested).
  • Replace the dddw_* drop-downs with autocomplete/typeahead components backed by paginated API endpoints.

FA integration

  • The current model where the Payroll module synchronously posts vouchers into the FA module is brittle — a transient FA error rolls back the whole payroll save.
  • Either make the FA posting an event published to a queue (RabbitMQ / Kafka / SNS) and let the FA service subscribe, or move both modules behind one transactional database where the same SQL transaction inserts the payroll row + the voucher rows.
  • Keep the df_fa_post "posting profile" table intact — it is genuinely useful and lets non-developers re-map accounts when the chart of accounts changes.

Reports

  • The 30 report DataWindows in FASPYW2.PBL are pixel-perfect statutory forms (Form 3A, 6A, 12A, Form 6, Form III). When you rebuild these, generate them with a server-side PDF library (e.g. wkhtmltopdf, Puppeteer, ReportLab) using the same field positions as the originals so users recognise them.
  • EPFO and ESIC have moved to electronic returns (ECR for EPF, monthly contribution file for ESIC). Add an "Export ECR" button alongside each form.

Phased rebuild plan

  1. Recover the data first. Extract FASMANGO.DB from the zip, install SQL Anywhere 5/6 (or use the SQLAnywhere ODBC driver), and dump every table to CSV. Even if you abandon the application, this preserves your father’s production data.
  2. Re-create the schema in PostgreSQL using the column listing in Section 7 of this document (corrected for modern types, with FK constraints, with timestamps, with proper enums).
  3. Load the CSV data into the new schema, mapping _ix decimal(0) values straight into BIGSERIAL / BIGINT columns and resetting the sequence high-water mark afterwards.
  4. Build the masters CRUD (Company, Department, Employee, Period, Leave). One generic CRUD component handles all of them; only the schema differs.
  5. Build the transaction screens (Attendance, Advance, Payment, Receipt) — same component, different validation rules.
  6. Re-implement wf_calc_pay as a pure function with unit tests using the actual recovered data as the test fixture (cross-check that net-pay numbers match the historical ti_py_pay rows for known periods).
  7. Build statutory reports last — they are read-only views on the calculated data.
