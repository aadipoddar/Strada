Target: Strada — Fleet Management software for Ashok Roadlines (.NET 10, Blazor Server + MAUI Hybrid, SQL Server/SSDT, Dapper, Syncfusion)
Requirement source: FAS Payroll (PowerBuilder 5.0 legacy system, C:\Dev\payroll) — full Indian payroll cycle

Date: 29 May 2026
What changed from the earlier draft and why. The first plan stored every statutory rate as a fixed column on PayrollDepartment (≈30 columns) and the per-employee applicability as boolean columns on Employee (IsEPFApplicable…). That works but every new scheme (NPS, a new cess, gratuity) needs a schema change — the same rigidity that trapped the legacy system with its 9 hardcoded salary components. This revision normalizes all statutory configuration into four small tables (StatutoryScheme, StatutoryRate, StatutorySlab, EmployeeStatutoryApplicability) plus a registration table. The payslip likewise stores statutory results as rows (PayslipStatutory), not columns. Result: adding a scheme = inserting data; the schema and the calculations stay stable. Everything downstream (engine, calculators, FA posting, reports) is rewritten here to match, so the plan is internally consistent end-to-end.

3. Data model overview & design principles

The model is built on five principles that keep it simple, correct, and future-proof:

#PrincipleHow it shows up
1Definitions are data, not columnsSalary components and statutory schemes are rows in master tables, not hardcoded columns. New component or scheme = new row.
2Effective-datingRates (StatutoryRate, EmployeeSalaryComponent) carry EffectiveFrom/To; the engine picks the row valid on the period date, so old payslips stay reproducible.
3Separate definition from resultComponent/scheme definitions live in masters; calculated amounts live in Payslip* rows. Changing a rate never alters a historical payslip.
4Results are rows, not columnsPayslipComponent (earnings/deductions) and PayslipStatutory (one row per scheme) mean the payslip never needs new columns when a scheme is added.
5Immutable runsA PayrollRun moves Draft→Approved→Posted→Paid; once posted, payslips and their vouchers are locked. Corrections flow forward as arrears.

3.2 The statutory model in one picture

StatutoryScheme says what a scheme is and which calculator handles it (EPF, ESI, PT, LWF…).
StatutoryRate says how much — employee %, employer %, wage ceiling, min/max, flat amounts — effective-dated. National schemes (EPF/ESI) use one all-India row; state schemes (LWF) are keyed by StateUTId; a company/department row overrides either only when needed.
StatutorySlab holds bands for slab schemes (Profession Tax), one set per state.
EmployeeStatutoryApplicability says who a scheme applies to (replacing the old per-scheme boolean columns) and carries the employee's registration number and any voluntary rate override.
The calculation engine reads these at run time; nothing statutory is hardcoded.

FlatProRataFixed monthly amount, reduced for unpaid days (Basic, DA, most allowances)round(rate × payableDays / periodDays)
FlatFixed amount, never pro-rated (e.g. Tiffin)rate

4.6 StatutoryRate NEW — effective-dated, scoped (national / state / company)

The numbers, as data. Each row carries a scope: leave StateUTId, CompanyId and PayrollDepartmentId all NULL for a national rate (EPF, ESI — identical everywhere in India, seeded once); set StateUTId for a state rate (e.g. LWF, which differs by state); set CompanyId/PayrollDepartmentId only to override one establishment. The engine reads the effective row whose scope is the most-specific match for the employee on the period date.

CREATE TABLE [dbo].[StatutoryRate]
(
    [Id] INT NOT NULL PRIMARY KEY IDENTITY,
    [StatutorySchemeId] INT NOT NULL,
    [StateUTId] INT NULL,                 -- NULL = national (EPF/ESI); set = state-specific (LWF...)
    [CompanyId] INT NULL,                 -- set only to override one company
    [PayrollDepartmentId] INT NULL,       -- set only to override one department
    [EmployeeRate] DECIMAL(9,4) NULL,     -- % (e.g. 12.0000, 0.7500)
    [EmployerRate] DECIMAL(9,4) NULL,     -- % (e.g. 3.6700, 8.3300, 3.2500)
    [WageCeiling] DECIMAL(18,2) NULL,     -- base cap (EPF 15000) / eligibility limit (ESI 21000)
    [MinAmount] DECIMAL(18,2) NULL,       -- e.g. EPF admin min 500 (establishment level)
    [MaxAmount] DECIMAL(18,2) NULL,       -- e.g. EPS max 1250
    [FlatEmployeeAmount] DECIMAL(18,2) NULL,  -- LWF employee
    [FlatEmployerAmount] DECIMAL(18,2) NULL,  -- LWF employer
    [ContributionMonths] VARCHAR(20) NULL,    -- non-monthly schemes, e.g. '6,12' for LWF; NULL = every month
    [EffectiveFrom] DATETIME NOT NULL,
    [EffectiveTo] DATETIME NULL,
    [Status] BIT NOT NULL DEFAULT 1,
    CONSTRAINT [FK_StatutoryRate_ToScheme] FOREIGN KEY ([StatutorySchemeId]) REFERENCES [StatutoryScheme]([Id]),
    CONSTRAINT [FK_StatutoryRate_ToStateUT] FOREIGN KEY ([StateUTId]) REFERENCES [StateUT]([Id]),
    CONSTRAINT [FK_StatutoryRate_ToCompany] FOREIGN KEY ([CompanyId]) REFERENCES [Company]([Id]),

)

Statutory rate resolver (used by the engine)

Picks the effective, most-specific rate for a scheme — a department/company override beats a state row, which beats the national row:

CREATE PROCEDURE [dbo].[Load_StatutoryRate_Effective]
    @StatutorySchemeId INT, @StateUTId INT, @CompanyId INT, @PayrollDepartmentId INT, @OnDate DATETIME
AS
BEGIN
    SELECT TOP 1 *
    FROM [dbo].[StatutoryRate]
    WHERE [StatutorySchemeId] = @StatutorySchemeId
      AND [Status] = 1
      AND @OnDate >= [EffectiveFrom]
      AND (@OnDate <= [EffectiveTo] OR [EffectiveTo] IS NULL)
      AND ([StateUTId] = @StateUTId OR [StateUTId] IS NULL)
      AND ([CompanyId] = @CompanyId OR [CompanyId] IS NULL)
      AND ([PayrollDepartmentId] = @PayrollDepartmentId OR [PayrollDepartmentId] IS NULL)
    ORDER BY CASE WHEN [PayrollDepartmentId] IS NOT NULL THEN 0 ELSE 1 END,  -- dept override first
             CASE WHEN [CompanyId] IS NOT NULL THEN 0 ELSE 1 END,            -- then company override
             CASE WHEN [StateUTId] IS NOT NULL THEN 0 ELSE 1 END,            -- then state, then national
             [EffectiveFrom] DESC;
END

The matching Load_StatutorySlab_Effective picks a state's Profession-Tax band for the base amount (and gender/month only where a state needs it):

CREATE PROCEDURE [dbo].[Load_StatutorySlab_Effective]
    @StatutorySchemeId INT, @StateUTId INT, @BaseAmount DECIMAL(18,2),
    @Gender VARCHAR(10), @Month INT, @OnDate DATETIME
AS
BEGIN
    SELECT TOP 1 *
    FROM [dbo].[StatutorySlab]
    WHERE [StatutorySchemeId] = @StatutorySchemeId
      AND [StateUTId] = @StateUTId
      AND [Status] = 1
      AND @OnDate >= [EffectiveFrom]
      AND (@OnDate <= [EffectiveTo] OR [EffectiveTo] IS NULL)
      AND @BaseAmount >= [FromAmount]
      AND (@BaseAmount <= [ToAmount] OR [ToAmount] IS NULL)
      AND ([Gender] = @Gender OR [Gender] IS NULL)
      AND ([ApplicableMonth] = @Month OR [ApplicableMonth] IS NULL)
    ORDER BY CASE WHEN [ApplicableMonth] IS NOT NULL THEN 0 ELSE 1 END,
             CASE WHEN [Gender] IS NOT NULL THEN 0 ELSE 1 END,
             [EffectiveFrom] DESC;
END

10. Calculation engine (standards-compliant)

A pure static method in StradaLibrary/Payroll/Calculation/PayrollCalculationEngine.cs — no DB calls, fully unit-testable. It takes the employee, attendance, component definitions+rates, applicable schemes and the effective statutory rates, and returns the payslip rows.

10.1 Order of operations (the standard sequence)

  1. Compute payable days = PeriodDays − LossOfPayDays.
  2. Compute each earning by its CalculationType (Percentage components resolve after their reference).
  3. Sum gross earnings; derive the per-scheme bases (EPF base, ESI base, PT base) from component flags.
  4. Run each applicable statutory calculator (§11), producing PayslipStatutory rows.
  5. Deductions = employee-side statutory + deduction-type components + income tax + advance instalments.
  6. Net = gross − deductions + other adjustment; optionally round to the rupee.

10.2 Core code

public static PayslipResult Calculate(PayslipInput in)
{
    var r = new PayslipResult();
    decimal periodDays  = in.Period.NumberOfDays;
    decimal payableDays = Math.Max(0, periodDays - in.Attendance.LossOfPayDays);
    decimal factor      = periodDays > 0 ? payableDays / periodDays : 0;
    r.PayableDays = payableDays;

    // ---- Earnings (Percentage components computed after slot-1 base exists) ----
    decimal gross = 0, epfBase = 0, esiBase = 0, ptBase = 0;
    var amountBySlot = new Dictionary<int, decimal>();

    foreach (var c in in.Components.OrderBy(c => c.SlotNo))
    {
        decimal rate = in.RateFor(c.Id);                    // employee rate or component default
        decimal amt = c.CalculationTypeCode switch
        {
            "Flat"        => rate,
            "FlatProRata" => Math.Round(rate * factor, 0, MidpointRounding.AwayFromZero),
            "PerDay"      => Math.Round(rate * payableDays, 0, MidpointRounding.AwayFromZero),
            "PerUnit"     => Math.Round(rate * in.UnitsFor(c.Code), 0, MidpointRounding.AwayFromZero),
            "Percentage"  => Math.Round(amountBySlot[c.ReferenceSlotNo.Value] * rate / 100m, 0, MidpointRounding.AwayFromZero),
            _             => 0m
        };
        amountBySlot[c.SlotNo] = amt;
        r.Components.Add(new(c.Id, c.SlotNo, c.Name, c.ComponentTypeId, rate, amt));

        if (c.ComponentTypeCode == "Earning" && c.IsPartOfGross) gross += amt;
        else if (c.ComponentTypeCode == "Deduction") r.OtherDeductions += amt;
        if (c.IsPartOfEPFBase)  epfBase += amt;
        if (c.IsPartOfESIBase)  esiBase += amt;
        if (c.IsPartOfPTaxBase) ptBase  += amt;
    }
    r.TotalEarnings = gross;

    // ---- Statutory (each calculator reads the effective StatutoryRate rows) ----
    var ctx = new StatutoryContext(in, gross, epfBase, esiBase, ptBase, payableDays);
    foreach (var scheme in in.ApplicableSchemes)
    {
        var line = StatutoryCalculatorFactory.For(scheme.CalculatorTypeCode).Calculate(scheme, ctx);
        if (line is null) continue;                          // not applicable this month
        r.Statutory.Add(line);
        if (scheme.IsEmployeeDeduction) r.TotalEmployeeStatutory += line.EmployeeAmount;
        r.TotalEmployerCost += line.EmployerAmount;
    }

    // ---- Income tax (manual), advance recovery, totals ----
    r.IncomeTax = in.Attendance.IncomeTaxDeduction;
    foreach (var a in in.OutstandingAdvances.Where(a => !a.SuspendDeduction))
    {
        decimal rec = Math.Min(a.InstalmentAmount, a.AdvanceAmount - a.TotalAdjusted);
        if (rec > 0) { r.AdvanceDeduction += rec; r.Adjustments.Add(new(a.Id, rec)); }
    }
    r.TotalDeductions = r.TotalEmployeeStatutory + r.OtherDeductions + r.IncomeTax + r.AdvanceDeduction;

    decimal net = gross - r.TotalDeductions + in.Attendance.OtherAdjustment;
    if (in.Department.RoundNetPayToRupee)
    {
        decimal rounded = Math.Round(net, 0, MidpointRounding.AwayFromZero);
        r.OtherAdjustment = in.Attendance.OtherAdjustment + (rounded - net);
        net = rounded;
    }
    else r.OtherAdjustment = in.Attendance.OtherAdjustment;
    r.NetPay = net;
    return r;
}
Net-pay rule (important): only employee-side statutory amounts (EPF employee 12%, ESI employee 0.75%, PT, LWF employee) reduce net pay. Employer EPF/EPS/EDLI/admin and employer ESI are employer cost (CTC) — stored on the payslip and posted to FA, but never subtracted from the employee's salary.

11. Statutory calculators (read from StatutoryRate)

Each StatutoryCalculatorType (its Code) maps to one strategy implementing IStatutoryCalculator. The strategy reads the effective StatutoryRate/StatutorySlab row(s) from the context and returns a PayslipStatutory line. Adding a scheme that fits an existing type needs no new code — only data. The context also exposes the employee's state, gender and the period month (derived from the company, employee and period), so state-specific schemes like PT and LWF resolve with nothing hardcoded.

11.1 EPF family (key EPF) — handles EPF, EPS, EDLI, EPF_ADMIN

All four share the EPF base (Basic+DA) capped at the wage ceiling. The single calculator branches on the scheme code:

var rate = ctx.Rate(scheme.Id);                       // effective StatutoryRate row
decimal cappedBase = rate.WageCeiling is decimal c ? Math.Min(ctx.EpfBase, c) : ctx.EpfBase;

decimal emp = 0, employer = 0;
switch (scheme.Code)
{
    case "EPF":                                       // employee 12% + employer 3.67%
        emp      = Round0(cappedBase * (ctx.OverrideRate(scheme.Id) ?? rate.EmployeeRate.Value) / 100);
        employer = Round0(cappedBase * rate.EmployerRate.Value / 100);
        break;
    case "EPS":                                       // employer 8.33%, capped (max 1250)
        employer = Round0(cappedBase * rate.EmployerRate.Value / 100);
        if (rate.MaxAmount is decimal m) employer = Math.Min(employer, m);
        break;
    case "EDLI":                                      // employer 0.50% (max 75)
        employer = Round0(cappedBase * rate.EmployerRate.Value / 100);
        if (rate.MaxAmount is decimal me) employer = Math.Min(employer, me);
        break;
    case "EPF_ADMIN":                                 // employer 0.50% (establishment min applied in summary)
        employer = Round0(cappedBase * rate.EmployerRate.Value / 100);
        break;
}
return new PayslipStatutoryLine(scheme, cappedBase, emp, employer);
The EPF-admin minimum ₹500 is an establishment-level floor, not per employee — applied once when building PayslipSummary (take MAX(sum of per-employee admin, 500)).

11.2 ESI (key ESI)

if (ctx.Gross > rate.WageCeiling) return null;        // not applicable when gross > 21000
decimal emp      = Math.Ceiling(ctx.Gross * rate.EmployeeRate.Value / 100);   // round UP to rupee
decimal employer = Math.Ceiling(ctx.Gross * rate.EmployerRate.Value / 100);
return new PayslipStatutoryLine(scheme, ctx.Gross, emp, employer);

ESI eligibility is normally frozen for the contribution period (Apr–Sep, Oct–Mar) even if wages cross the limit mid-period; handle that by checking eligibility at the period's contribution-period start (a flag carried on the context).

11.3 Profession Tax (key SLAB)

// slab resolved for the employee's state (and gender/month only where a state needs it)
var slab = ctx.Slab(scheme.Id, ctx.StateUTId, ctx.PtBase, ctx.Gender, ctx.Month);
decimal emp = slab?.Amount ?? 0;
return new PayslipStatutoryLine(scheme, ctx.PtBase, emp, 0);   // PT is employee-only; a state with no PT -> 0

11.4 LWF (key FLAT)

// contribution months come from the rate row (e.g. '6,12'), so each state's schedule is data
if (!ctx.IsContributionMonth(rate)) return null;
return new PayslipStatutoryLine(scheme, 0, rate.FlatEmployeeAmount ?? 0, rate.FlatEmployerAmount ?? 0);
Why this is simpler than the legacy code. The old system had nine near-identical wf_calc_comp_* functions plus hardcoded slabs and a hardcoded 2002 ESI date. Here there are four small calculators, the slabs and rates are data, and a new scheme like NPS that is “percentage of base” reuses an existing key with zero new code.

20. Final verdict

This plan is coherent, standards-compliant, and simpler where it counts. Normalizing statutory configuration into StatutoryScheme / StatutoryRate / StatutorySlab / EmployeeStatutoryApplicability — and storing results as PayslipComponent + PayslipStatutory rows — removes the two rigidities that would otherwise force schema changes (new scheme, new component) and matches how SAP, Oracle and Zoho are built.

The calculations were re-derived against current (2025–26) statutory rules and traced in a full worked example (net ₹15,570, employer cost ₹2,015 on ₹17,173 gross). The engine is a small, pure, testable function; the four calculators read their numbers from data; the payroll run is immutable once posted and balanced into FA.

Non-negotiables when building: employee vs employer split (employer EPS/EDLI/admin/ESI never reduce net pay); DECIMAL not MONEY; ESI rounds up; EPF base capped at the ceiling; rates resolved by effective date; golden-master tests against the legacy data first. Build P1→P4 in order and the module will be correct, compliant, and easy to keep current as the law changes.

Plain-English glossary — read this first. The app uses short government names; here is every one in simple words.
  • Gross = everything earned before any cut. Deduction = money cut from salary. Net pay / take-home = what the employee actually gets (gross − deductions). Employer cost = extra the company pays on top of salary — it never reduces take-home.
  • Basic = core salary. HRA = House Rent Allowance. Conveyance = travel allowance. Tiffin = food allowance. (These are just names of "pay parts".)
  • EPF/PF = Provident Fund, a retirement saving (12% cut from employee, company adds its share). EPS = the pension part of PF (company-only). EDLI = a small PF life-insurance (company-only). EPF Admin = PF processing fee (company-only).
  • ESI = Employees' State Insurance, medical cover for lower-paid staff (only if monthly gross ≤ ₹21,000). PT = Profession Tax, a small state tax on salary (varies by state). LWF = Labour Welfare Fund, a tiny state contribution a few times a year. TDS / Income Tax = income tax held back from salary.
  • LOP = Loss Of Pay = unpaid absent days. Advance = a loan to the employee, recovered in monthly instalments. UAN = employee's PF number; ESI IP no. = employee's ESI number; PAN = tax ID; IFSC = bank-branch code.
  • RoundNetPayToRupee = a yes/no setting: if yes, take-home is rounded to whole rupees (₹15,569.60 → ₹15,570) and the few paise are recorded as a tiny adjustment.
  • WageCeiling = the most salary a rate is charged on (PF is figured on at most ₹15,000). Min/Max = a floor/cap on the resulting amount. EffectiveFrom = the date a rate starts (old payslips keep using the old rate).
  • CalculatorType = which built-in formula a scheme uses (you never change it). Scope = where a rate applies: National (all India), State (one state), or an override for one company.
How to read the tables below.
  • A 1 means yes, 0 means no — these are on/off switches on the screen.
  • A value with slashes (e.g. 1 / 1 / 1 or 3 / 15) is just several columns squeezed into one cell to save space; each part is its own field, explained where it appears. You do not type slashes — on the screen each is a separate box/toggle.
  • A middot (e.g. BASIC 12000 · HRA 40) separates what are really separate rows.
  • Where a column is a fixed list (Type, CalculationType, RunType, State, Mode of payment…), the tables show the readable name, but the database stores that list's Id — the small lookup tables are in §4.12, and the ones used here are listed first in 21.2.
  • Three kinds of data: things that ship with the product (the schemes & rates), things you type in (employees, amounts, attendance), and things the app calculates for you (the payslips & the accounting entry).
